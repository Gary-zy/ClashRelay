# AI开发守则

## 第一原则

后续所有 AI 开发必须把 `specs/` 当成项目规范源。改代码不更新对应 specs，等于任务没完成。

## 开发前检查

开始任何代码改动前：

1. 读 `specs/README.md`。
2. 读本文件。
3. 按任务类型读取相关规范：
   - 协议解析、YAML 输出、规则：`代理协议与配置生成规范.md`
   - UI、交互、文案：`前端交互规范.md`
   - 模块拆分、重构：`架构设计.md`、`模块职责.md`
   - 表单字段、localStorage、节点字段：`数据模型与状态.md`
   - 服务端、命令、部署：`运维手册.md`
4. 检查 `git status --short`，不要回滚用户已有改动。

## 开发中约束

- 优先复用现有模块和测试方式。
- 能写纯函数就不要把逻辑塞进 Vue 模板或事件里。
- 能用 `js-yaml`、URL、URLSearchParams 等结构化 API，就不要手搓字符串解析。
- 配置生成必须避免悬空引用。
- 网络入口必须考虑安全边界。
- 错误提示必须可操作。
- 新增行为必须配测试；文档改动可不跑全量测试，但要做文档自检。

## Specs 同步规则

| 改动类型 | 必须更新 |
|----------|----------|
| 新增协议、改协议字段 | `代理协议与配置生成规范.md` |
| 改策略组、默认规则、YAML 结构 | `代理协议与配置生成规范.md` |
| 改工作台步骤、按钮、导入路径 | `前端交互规范.md` |
| 改表单字段、节点字段、localStorage key | `数据模型与状态.md` |
| 新增模块、移动职责、重构边界 | `架构设计.md`、`模块职责.md` |
| 改 npm scripts、端口、环境变量、Docker/Tauri/mihomo | `运维手册.md` |
| 改产品模式或用户场景 | `项目总览.md` |
| 发现 specs 与代码不一致 | 先核实，再修对应 specs 或代码 |

## 完成前清单

提交或回复完成前，必须确认：

- 相关测试已运行，或明确说明为什么没运行。
- `specs/` 是否需要更新已经判断。
- 如果需要更新 specs，已经同步更新。
- 没有留下 `TODO`、`TBD`、空章节。
- 没有把用户未提交改动回滚掉。
- 最终说明里写清楚改了哪些代码、哪些 specs、跑了哪些验证。

## 给后续 AI 的硬提示

如果用户让你“继续开发”“加功能”“修 bug”“改 UI”“改配置生成”，不要只看代码。先看 specs。RelayBox 后续维护靠这套规范续命，别整那种今天能跑、明天没人敢改的活儿。
